# 事实核查协议详解

刘备教授体的"信任感"建立在每段都有可验证的具体事实之上。**事实写错 = 文章失败，比文风不像更严重**。这份文件给你具体的判断流程和处理范例。

---

## 一、为什么这件事比风格更重要

刘备教授体本身就是一种"我和你一样在看新闻"的同伴感。作者每写一条编号短评几乎都带数字（如"宁德 Q3 净利 305 亿"、"美光跌 10cm"、"小米锁单 1.5 万"），读者会自然相信他在"陪你看市场"。

如果模型瞎编"昨天小米锁单 50 万"，读者一查发现根本没这事，整篇文章的信任就崩了——而且写得越像刘备教授，崩得越彻底（因为读者本来期待真实数据）。

**这是 skill 设计上的硬约束**：宁可花一轮对话问清楚，也不要乱编。

---

## 二、识别模糊事实的清单

读完用户主题后，按这些类别逐项扫描：

### 2.1 人物状态类（高风险）

| 模糊词 | 可能解读 | 必须确认 |
|---|---|---|
| "XX 走了" | 卸任/下台/去世/离开会议/出国/离婚 | 是 |
| "XX 出事了" | 丑闻/被抓/被诉/破产/事故 | 是 |
| "XX 换人了" | 谁卸任/谁上任 | 是（两侧都要） |
| "XX 翻车了" | 经营/丑闻/产品/股价 | 是 |
| "XX 凉了" | 公司倒闭/项目失败/失势 | 是 |
| "XX 复活了" | 业务回暖/恢复职位/平反 | 是 |

### 2.2 事件细节类（中风险）

| 模糊词 | 可能解读 | 必须确认 |
|---|---|---|
| "昨天 / 最近 / 前几天 / 上周" | 具体日期 | 是（结合系统当前日期） |
| "XX 公司又出大事" | 哪家？什么事？ | 是 |
| "那个新闻" | 具体哪条 | 是 |
| "现在那个谁" | 具体哪个人 | 是 |

### 2.3 数字类（中风险）

| 模糊词 | 可能解读 | 处理 |
|---|---|---|
| "涨了 / 跌了"但没数字 | 涨跌幅多少 | 查盘面数据或问 |
| "大涨 / 暴跌" | 一般 >5%，但要确认 | 查 |
| "创新高 / 创新低" | 历史新高还是阶段新高 | 查 |
| "差不多 / 大概多少" | 具体范围 | 查或问 |

### 2.4 绰号映射类（低风险，但偶尔出错）

用户用的绰号一般都对应明确的人，但偶尔会有歧义：

- "马哥" → 是马云还是马化腾还是马斯克？
- "老黄" → 是黄仁勋还是黄峥（拼多多）？
- "X 总" → 中文里太常见，可能歧义

如果上下文明确就直接用，不明确就在心里默认 + 在文章中用全名首次出现，避免读者搞混。

## 三、置信度判断方法

对每个事实锚点问自己三个问题：

1. **训练数据里有吗？** — 例如 "2024 年特朗普赢了大选" 是高置信度，"2026 年 X 月特朗普签了 Y 法案" 可能是低置信度。
2. **是否有时效性？** — 股价、汇率、官员任命这种是高时效性，必须查最新。
3. **有几种可能解读？** — 用户用模糊词时，列出 3+ 种可能解读，如果有 2 种以上"明显合理"，必须问用户。

**判断快捷法**：如果你下笔写之前，心里有"我猜应该是 X 吧"的犹豫——那就是低置信度，必须查或问。

## 三·五、WebSearch 摘要 ≠ 事实（核查工具使用规范）

这条规则源于一次真实失败：用 WebSearch 查 A 股 5/15 收盘，AI 摘要给出"涨 0.52% 报 4106.26"——结果是把"周五前瞻文章里的预测"和"另一时段数据"拼到一起的错误合成，实际当天上证跌 1.02% 报 4135.39。用户一眼看穿，文章信任崩塌。

### 工具的真实工作机制

| 工具 | 返回的是什么 | 可信度 |
|---|---|---|
| **WebSearch** | 搜索引擎的链接列表 + AI 合成的"总结段" | 链接是真实的；总结段是 AI 拼接，**对具体数字尤其不可靠** |
| **WebFetch** | 实际打开链接、读取原文、按你的 prompt 提取 | 内容是真的网页内容，可信度高 |
| **AskUserQuestion** | 直接问用户 | 最可信，但代价是打断用户 |

### 硬规则

1. **WebSearch 的 AI 摘要不能直接作为事实引用**。它只能用于：
   - 找到相关链接（用 WebFetch 进一步读）
   - 判断事件大方向（XX 发生了 / 谁上任了 / 哪天发生的）
   - 不能用于：**具体数字（涨跌幅、点位、价格、营收、票数）**

2. **遇到具体数字必须用 WebFetch 读原文**。例如：
   - 不能写 "上证跌 1.02% 报 4135.39"（如果只是 WebSearch 摘要给的）
   - 应该 WebFetch 打开来源链接，读到正文里的实际数字，再写

3. **WebFetch 读不到时（动态加载/付费墙/被拒绝）→ 用范围词**：
   - ✓ "上证小跌"、"跌了 1cm 多"、"跌了 1cm 左右"
   - ✗ "上证跌 1.02% 报 4135.39"（凑的精确数字）

4. **两次搜索结果矛盾 = 默认不可信**：
   - 一次说"涨 0.52%"，另一次说"跌 1.02%" → 不能挑一个用
   - 必须换 query、换源、用 WebFetch 读原文，或问用户

5. **判断"够不够准"的快捷法**：你打算引用的数字，能不能告诉读者"这个数字我从 XX 网站某篇文章的第 X 段读到"？答得上来 = 真核查了。答不上来 = WebSearch 摘要而已，不能用。

### 标准核查流程（升级版）

遇到模糊事实时，按这个顺序走：

```
1. WebSearch 找方向
   ↓
   是不是大方向能定？（事件、人物、日期）
   ↓
2. 锁定关键链接
   ↓
   WebFetch 打开来源读原文
   ↓
3. 原文有具体数字？
   ↓
   ├─ 有 → 用原文数字
   └─ 没/读不到 → 用范围词 或 AskUserQuestion 问用户
```

**禁止跳过第 2 步直接用 WebSearch 摘要**。这是上一次失败的根因。

## 四、处理流程的范例

### 范例 1：用户说"昨天特朗普走了，美联储主席换人了，A 股继续跌"

**第 1 步：识别事实锚点**
- "昨天" → 是哪天？（今天的日期 - 1）
- "特朗普走了" → 4 种可能解读（卸任/去世/离开会议/出国访问）
- "美联储主席换人了" → 谁卸任？谁上任？
- "A 股继续跌" → 跌多少？哪些板块？

**第 2 步：判断置信度**
- "昨天" → 高（可以从系统日期推算）
- "特朗普走了" → 低（多种合理解读）
- "美联储主席换人" → 低（如果模型不知道具体任命）
- "A 股继续跌" → 中（需要查最新数据）

**第 3 步：执行**
1. 用 WebSearch 查最近 48 小时关于"特朗普"+"美联储主席"的新闻。
2. 如果搜到明确答案 → 直接用。
3. 如果搜不到或仍有歧义 → 用 AskUserQuestion：

```
问题 1："昨天特朗普走了" 具体指什么？
- 卸任/下台（提前结束总统任期）
- 去世
- 访问外国/离开美国
- 离开某个会议/活动
- 其他（用户填写）

问题 2：美联储主席换人了，是指鲍威尔卸任、新主席上任了吗？新主席是谁？
- 鲍威尔卸任，沃什上任
- 鲍威尔卸任，米兰上任
- 鲍威尔卸任，其他人上任（用户填写）
- 是传闻不是事实，按假想情景写

问题 3：A 股昨天跌幅是多少？是大盘跌还是某些板块跌？
- 大盘跌 1-2cm，全板块普跌
- 大盘跌 3cm 以上，恐慌情绪
- 大盘小跌，但某板块（科技/消费/金融）大跌
- 我也不知道具体数字，你查最近数据
```

**第 4 步：拿到用户回答后**

按用户给的事实写文章，把核查过的事实标在心里清单上：
- 川普 = X（用户确认）
- 新美联储主席 = Y（用户确认 或 假想情景）
- A 股跌幅 = Z（用户给或自己查）

---

### 范例 2：用户说"今天黄金又创新高，写一篇"

**第 1 步：识别事实锚点**
- "今天" → 系统日期
- "黄金创新高" → 多少美元/盎司？是否真的创历史新高？

**第 2 步：判断置信度**
- "今天" → 高
- "黄金创新高" → 中（大方向对，但具体价位要查）

**第 3 步：执行**
WebSearch 查"黄金价格 今天 创新高" → 拿到具体价格写文章。

不需要打断用户问。这种情况比 范例 1 简单很多——主题清晰、事实方向明确、只是数字需要核查。

---

### 范例 3：用户说"小米最近 SU7 卖得不错，我看一些八卦说雷总要造游艇了"

**第 1 步：识别事实锚点**
- "SU7 卖得不错" → 销量数字
- "雷总要造游艇" → 真的吗？还是网络段子？

**第 2 步：判断置信度**
- "SU7 销量" → 中，需要查最新数据
- "雷总造游艇" → 低，明显需要核实（这听起来像段子）

**第 3 步：执行**
1. WebSearch 查"雷军 游艇" + "小米 SU7 销量 最新"。
2. 如果"雷总造游艇"查不到任何来源 → 问用户：

```
问题：你提到"雷总要造游艇"——这是真的新闻还是网上的段子？
- 这是最近的真新闻，请帮我查一下
- 是网上流传的段子/玩笑，按段子调侃写
- 我看错了/记错了，跳过这个点
```

**记住**：刘备教授会调侃段子（"东哥你是决定不烧外卖了么"），但调侃前都知道是段子。模型如果把段子当真事写，就会出洋相。

---

### 范例 4：用户说"用刘备教授风格写一篇关于明天 A 股的展望"

**第 1 步：识别事实锚点**
- "明天 A 股" → 明天还没发生，无法核查
- "展望" → 预测性内容

**第 2 步：判断**
这是个"前瞻"主题，不需要核查"明天发生了什么"（因为还没发生）。但需要核查的是：
- **当前** A 股的位置、情绪、关键事件
- **当前** 影响因子（美联储动态、地缘事件、政策预期）

**第 3 步：执行**
查最近 24 小时的市场动态 + 最近一周的关键事件，然后基于这些事实做"展望"。

写作时大量用概率词（大概率/小概率/三七开）——这本来就是作者写展望的风格。

## 五、AskUserQuestion 使用规范

### 5.1 何时用 AskUserQuestion

- 一个模糊点有 3+ 种合理解读
- 用户没给关键事实但文章必须用
- WebSearch 结果矛盾或不足

### 5.2 何时不用 AskUserQuestion

- WebSearch 已经给出清晰答案 → 直接用
- 用户给的素材已经足够明确 → 直接写
- 文章里这个点其实可以不出现 → 跳过

### 5.3 选项设计原则

- 一个问题给 2-4 个选项，加上系统自动添加的"其他"
- 选项是**互斥**的（用户能选且只选一个，除非 multiSelect=true）
- 选项描述要简短但完整，让用户能一眼判断
- 不要把"我不知道"作为选项——用户来找你就是因为他知道
- 不要让用户填长字段——选项应该比让用户打字更省力

### 5.4 一次问太多 vs 一次问太少

- 太多：5+ 个问题一次发，用户会烦
- 太少：一次问一个，需要 3 轮对话才能开始写，体验差
- **合适**：一次问 2-3 个相关问题（同一个事件的不同细节），用户能一波回完

如果模糊点超过 4 个，说明用户给的主题太抽象——可以先回复一句"主题有点抽象，能不能说说具体是哪条新闻让你想写这个？"，让用户先给个更具体的种子。

## 六、绝对禁止的事

1. **看到模糊事件就硬挑一个解读写**。例如"特朗普走了"自动写成"川普卸任后……"——这是 skill 设计上的反模式。
2. **用"可能"、"据说"、"传闻"等弱化词来掩盖事实不清**。例如"据说昨天特朗普卸任了，市场反应是……"——这是把模型自己的不确定推给读者，文风也不像（作者用"据说"是引用真有的传闻，不是掩盖事实不清）。
3. **写假新闻当真新闻**。用户没说"假想情景"时，所有事件都默认为真实事件。
4. **明知数字不准但凑出一个具体值**。例如"今天 A 股跌 2.34cm"——如果你没查最新数据，就别凑这种精确数字，用"跌了一些"或问用户。
5. **把 WebSearch 的 AI 摘要当事实引用**。WebSearch 返回的"总结段"是 AI 拼接的，对具体数字（涨跌幅、点位、价格、营收）尤其不可靠。引用具体数字前**必须**用 WebFetch 实际打开来源链接读原文。详见上一节"三·五"。
6. **两次搜索结果矛盾时挑一个用**。如果一次搜到"涨 0.52%"另一次搜到"跌 1.02%"，不能挑感觉对的那个用——这意味着两次都不可信。必须换 query、用 WebFetch 读原文、或问用户。

## 七、当事实核查发现用户提供的素材本身有错

如果 WebSearch 之后发现用户给的素材和现实不符（比如用户说"美联储主席换成 X 了"，但实际还是鲍威尔），按这个流程处理：

1. **先告知用户冲突**：
   > "我查了下，目前美联储主席还是鲍威尔，没有换人。你想：(A) 按事实写 鲍威尔还在任；(B) 这是假想情景，假设 X 已经上任；(C) 我记错了，跳过这个点。"

2. **不要默认按用户的素材硬写**——除非用户明确说"这是假想"或"我知道，假设这个发生了"。

3. **如果用户选 B（假想情景）**：可以写，但在文章开头一句话标记（"假设 X 上任的话……"），让读者知道这是推演而不是事实。

---

## 八、跟刘备教授真实文章的对照

回头看 269 篇样本，作者**几乎没有**事实硬错的地方。他偶尔会用"据说"、"传"、"听说"，但都是真实存在的传闻；他用绰号、戏谑、调侃，但底层事实都对。

模仿他的核心不是模仿"看起来很懂"，而是模仿"真的在认真看新闻"。事实核查协议就是把这种"真的在认真看新闻"的态度，硬编码到 skill 里。
